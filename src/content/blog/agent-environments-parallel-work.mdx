Останнім часом я тестував різні рішення, щоб комфортно працювати в агентному середовищі. Хотілось, щоб можна було запустити декілька сесій в одному репозиторії паралельно, і при цьому спокійно керувати сесіями в різних репозиторіях — тобто одночасно вести кілька задач чи проєктів і не сходити з розуму.

Ясна річ, для цього потрібна підтримка **git worktree** — інакше агенти будуть топтатись по одних і тих самих файлах, і замість паралельної роботи отримаєш паралельний хаос.

<Callout type="info" title="Що таке git worktree?">
  Це вбудована можливість git, яка дозволяє мати декілька робочих копій одного
  репозиторію в окремих папках — кожна на своїй гілці. Без stash, без
  перемикання гілок, без конфліктів. Ідеально для паралельних агентів: кожен
  працює у своїй копії і нікому не заважає.
</Callout>

## Варіант 1: Warp

Найпростіший варіант для старту — термінал, у якому підтримка git worktree є з коробки. Warp сам розпізнає worktree-копії, і все, що прив'язане до репозиторію (правила проєкту, контекст кодової бази, агенти), коректно працює у кожній з них.

<ProsCons>
  <Pros title="Плюси">
    - Вкладки гарно показують статуси сесій — одним поглядом видно, хто працює, хто чекає, хто завершив
    - Можна писати код прямо там — це все ж таки повноцінний термінал з редактором
  </Pros>
  <Cons title="Мінуси">
    - Поки не знайшов, як швидко видаляти створені копії. Worktree накопичуються, залишається багато сміття, яке жере пам'ять на диску. Треба не забувати чистити вручну — а я, звісно, забуваю
  </Cons>
</ProsCons>

## Варіант 2: Нативний застосунок Claude (режим Code)

Десктопний застосунок Claude з режимом Code — кожна нова сесія автоматично отримує свій worktree, без додаткових команд і ручних налаштувань.

<ProsCons>
  <Pros title="Плюси">
    - Вміє сам видаляти копії — а з GitHub взагалі автоматично прибирає worktree після злиття pull request. Ніякого сміття
    - Може працювати у хмарному режимі: можна ставити задачі з телефона, поки їдеш кудись чи стоїш у черзі. Щоправда, тільки для репозиторіїв на GitHub
    - Вміє підключатись через SSH до VPS. Може, не для щоденних робочих потреб, але все одно круто — можна попросити агента щось полагодити прямо на сервері
    - Є інтегрований браузер — агент відкриває сторінку, а ти можеш прямо в ній залишати коментарі до інтерфейсу
    - Може працювати і без worktree, прямо в оригінальному репозиторії, якщо задача маленька
  </Pros>
  <Cons title="Мінуси">
    - Писати код вручну не можна. Дерево файлів є, подивитись код можна, а редагувати — ні. Для мене це суттєво, бо іноді швидше самому поправити рядок, ніж пояснювати агенту
    - Термінал і дерево файлів можна відкрити тільки після запуску агента
  </Cons>
</ProsCons>

## Варіант 3: Conductor

Conductor — це вже повноцінне середовище мультиагентної розробки для Mac. Кожна задача отримує свій робочий простір: окремий worktree, гілку, термінал, перегляд змін (diff) і процес перевірки коду. Застосунок безкоштовний — використовуєш свою підписку Claude (або Codex).

<ProsCons>
  <Pros title="Плюси">
    - Сам створює і видаляє worktree — весь життєвий цикл під контролем: зробив задачу, злив зміни, заархівував робочий простір
    - Дуже приємні сповіщення про статуси — завжди знаєш, коли агент закінчив або чекає на тебе
    - Можна писати код прямо там
    - Інтерфейс найкращий з усіх варіантів — видно, що людям було не байдуже
    - Є зручна кнопка, щоб відкрити репозиторій в іншому застосунку: IDE, будь-який термінал, Finder
  </Pros>
  <Cons title="Мінуси">
    - Не працює без worktree. Дрібний недолік, але іноді хочеться просто швидко попрацювати в оригінальній папці
  </Cons>
</ProsCons>

<Callout type="tip" title="Бонус">
  Conductor копіює тільки ті файли, які відстежує git — тобто node_modules і
  .env не дублюються на кожен робочий простір. При цьому можна налаштувати,
  які саме env-файли треба копіювати в нові робочі простори. Дрібниця, а
  приємно.
</Callout>

## На чому зупинився

Поки що — на **Conductor**. Комбінація «повний життєвий цикл worktree + найкращий інтерфейс + можливість писати код самому» перекрила все інше.

Взагалі, просто працювати в IDE по-старому стає вже якось не дуже зручно. Робочий процес зміщується від «я пишу код, агент допомагає» до «агенти пишуть код, я оркеструю і перевіряю». І інструменти під це потрібні відповідні.

## P.S. про VS Code

Ще дуже крутий **VS Code в Agents Mode** — у них тепер є окреме вікно Agents, де можна запускати і локальних, і хмарних агентів паралельно. Інтерфейс топовий. Але є одне велике «але»: він не підтримує сторонні підписки чи акаунти — тільки GitHub Copilot. Навіть Claude-агенти там працюють через Claude Agent SDK, але оплата йде виключно через підписку Copilot, свій ключ чи акаунт підключити не можна.

Шкода, що не можна використовувати зі своїм акаунтом Claude. Якби можна було — можливо, цей пост мав би іншого переможця.
